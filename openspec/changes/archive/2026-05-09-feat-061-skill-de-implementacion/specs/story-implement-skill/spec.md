[[FEAT-061-skill-de-implementacion-el-programador-autonomo]]

## ADDED Requirements

### Requirement: story-implement skill exists in canonical location
The system SHALL provide the `story-implement` skill at `.claude/skills/story-implement/SKILL.md` with its `assets/` and `examples/` directories.

#### Scenario: Skill directory is present
- **WHEN** the repository is inspected
- **THEN** `.claude/skills/story-implement/SKILL.md` exists and contains valid YAML frontmatter with `name: story-implement`

### Requirement: Skill requires all three planning artifacts before implementing
The system SHALL verify that `story.md`, `design.md`, and `tasks.md` exist in the target story directory before executing any implementation task.

#### Scenario: design.md or tasks.md is missing
- **WHEN** `/story-implement` is invoked on a directory that lacks `design.md` or `tasks.md`
- **THEN** the skill halts immediately with an error message listing the missing artifacts and suggests running `/story-plan` first

## Requirement: Implement must comply with Project Policies and Definition of Done
The skill must adhere to the project policies defined in `$SPECS_BASE/policies/constitution.md` and `$SPECS_BASE/policies/definition-of-done.md`. These policies establish the technical principles, quality standards, and acceptance criteria that guide the design and implementation process. The skill must generate code that meets these standards and criteria, ensuring that the implementation not only works but is also maintainable, scalable, and aligned with the project's best practices. Any deviations from these policies must be documented in the final report generated by the skill upon completion of the implementation. The project policies and the Definition of Done are fundamental to ensuring that the code generated by the skill meets the project's quality requirements and expectations, providing a clear framework for AI-assisted autonomous implementation.

### Requirement: Skill implements each task following TDD
The system SHALL process each pending task in `tasks.md` in order, generating a failing test first and then the minimal production code to make it pass.

#### Scenario: Full TDD cycle for a single task
- **WHEN** the skill processes a pending task with all referenced components defined in `design.md`
- **THEN** the skill generates a test file for that task before generating the production code

#### Scenario: All tasks complete successfully
- **WHEN** all pending tasks in `tasks.md` are implemented without blockers
- **THEN** all task checkboxes are marked `- [x]` and `implement-report.md` is generated with status ✓ for each task

### Requirement: Skill updates tasks.md in real time
The system SHALL mark each task as complete (`- [ ]` → `- [x]`) or blocked (`- [ ]` → `- [~]`) immediately after processing it, without waiting until all tasks are done.

#### Scenario: Task marked complete immediately
- **WHEN** the skill finishes implementing a task
- **THEN** `tasks.md` is updated with `- [x]` for that task before the skill begins the next task

### Requirement: Skill handles unimplementable tasks without stopping the pipeline
The system SHALL detect when a task references a component not defined in `design.md`, mark that task as blocked, and continue with the remaining tasks.

#### Scenario: Task references undefined component
- **WHEN** a task description references a component name not found in `design.md`
- **THEN** the skill marks the task as `- [~]` (blocked), registers it in `implement-report.md` as "requiere aclaración", and proceeds to the next task

#### Scenario: Blocked tasks do not prevent report generation
- **WHEN** one or more tasks are blocked but all other tasks complete
- **THEN** `implement-report.md` is generated with all tasks listed, distinguishing completed (✓) from blocked (⚠️ requiere aclaración)

### Requirement: Skill generates implement-report.md on completion
The system SHALL generate `implement-report.md` in the story directory after processing all tasks, containing a traceability table by task ID and a list of blocked tasks with their reasons.

#### Scenario: Report generated after full pipeline
- **WHEN** all tasks (completed and blocked) have been processed
- **THEN** `implement-report.md` exists in the story directory with a row per task showing ID, description, status, and generated files

### Requirement: Skill warns when task count exceeds recommended limit
The system SHALL emit a warning when `tasks.md` contains more than 20 pending tasks and suggest splitting the story using `story-split`.

#### Scenario: Large task count detected
- **WHEN** the skill resolves a `tasks.md` with more than 20 pending tasks
- **THEN** the skill displays a warning before starting implementation and suggests running `/story-split` first
